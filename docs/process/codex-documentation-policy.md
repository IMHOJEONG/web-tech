# Codex Documentation Policy

## Purpose

이 문서는 Codex가 수행한 작업을 항상 추적 가능하게 남기기 위한 운영 규칙입니다.

핵심 원칙:

- 모든 작업은 코드 변경과 함께 문서 업데이트를 남깁니다.
- 문서는 다음 사람이 바로 이어받을 수 있을 정도로 짧고 명확해야 합니다.
- 구현 세부사항보다 의도, 범위, 영향, 다음 액션이 먼저 드러나야 합니다.

## Required Updates

Codex가 작업할 때 기본적으로 지켜야 하는 최소 문서화 범위는 다음과 같습니다.

1. 모든 작업은 `docs/worklog/`에 기록합니다.
2. 구조적 결정이나 장기 영향이 있는 변경은 `docs/architecture/`에 ADR로 남깁니다.
3. 반복될 운영 절차나 협업 규칙은 `docs/process/` 또는 향후 `docs/runbooks/`에 정리합니다.

## Worklog Rule

작업이 끝날 때 `docs/worklog/`에는 아래 항목이 남아 있어야 합니다.

- `Summary`
- `Changed`
- `Notes`
- `Open Questions`
- `Next`

최소 기준:

- 어떤 작업을 했는지
- 어떤 파일이 바뀌었는지
- 구현 중 확인한 제약이나 의사결정이 무엇인지
- 다음에 이어서 해야 할 일이 무엇인지

## ADR Rule

다음 조건 중 하나라도 해당되면 ADR 작성을 검토합니다.

- 페이지 구조나 앱 셸 구성이 바뀔 때
- 공용 컴포넌트 책임이 달라질 때
- 데이터 흐름이나 라우팅 기준이 달라질 때
- 이후 작업에 반복적으로 참조될 결정이 생길 때

## Practical Standard

문서화는 과하게 길게 쓰기보다 다음 기준을 만족하면 충분합니다.

- 작업 배경을 처음 보는 사람이 이해할 수 있다.
- 바뀐 범위와 이유를 1분 안에 파악할 수 있다.
- 다음 작업자가 바로 이어서 구현할 수 있다.

## Current Team Rule

현재 프로젝트에서는 다음 규칙을 기본값으로 사용합니다.

- Codex는 작업 완료 시마다 문서를 함께 갱신합니다.
- 특별한 요청이 없어도 `worklog` 업데이트를 생략하지 않습니다.
- 사용자가 "문서로 남겨달라"는 요청을 한 경우, 일회성 메모가 아니라 지속적으로 관리되는 문서 체계에 반영합니다.
- 사용자가 구조 선호를 명시한 경우
  - 예: barrel file 비선호
  - 이후 작업에서도 해당 선호를 기본값으로 존중합니다.
