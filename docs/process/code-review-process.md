# 코드 리뷰 프로세스

## 목적

이 문서는 이 프로젝트에서 코드 리뷰를 어떤 기준과 순서로 운영할지 정리합니다.

핵심 목표는 다음과 같습니다.

1. 회귀를 줄인다.
2. 설계 의도를 문서와 코드에 함께 남긴다.
3. 리뷰가 감상이나 취향 논쟁이 아니라, 위험 관리와 품질 점검이 되게 한다.

## 기본 원칙

- 리뷰는 `버그`, `회귀`, `보안`, `운영 리스크`를 먼저 본다.
- 취향보다 일관성을 우선한다.
- 구조적 변경은 코드와 함께 문서도 확인한다.
- 발견사항은 `심각도 순서`로 정리한다.
- 구현자와 리뷰어 모두 다음 사람이 이어받을 수 있는 상태를 만든다.

## 언제 리뷰가 필요한가

아래 중 하나라도 해당되면 코드 리뷰를 거친다.

- 사용자-facing UI가 바뀐다.
- 라우팅, 데이터 흐름, env, 인증 규칙이 바뀐다.
- `apps/docs` 또는 공용 패키지(`packages/*`)가 바뀐다.
- 빌드/배포/CI 동작에 영향이 있다.
- 문서 정책이나 운영 기준이 바뀐다.

작고 국소적인 수정이라도:

- 회귀 위험이 있거나
- 설계 의도가 애매하거나
- 이후 반복될 패턴을 건드리면

리뷰를 거친다.

## 리뷰 흐름

### 1. 작성자 셀프 체크

작성자는 리뷰 요청 전에 아래를 먼저 확인한다.

- 타입 체크
- 관련 테스트 또는 최소 실행 확인
- 문서 갱신 여부
- 불필요한 변경 섞임 여부
- env/비밀값 노출 여부

### 2. 변경 범위 요약

리뷰 요청 시 최소한 아래를 함께 남긴다.

- 무엇을 바꿨는가
- 왜 바꿨는가
- 어디를 보면 되는가
- 어떤 리스크가 남아 있는가

### 3. 리뷰어 확인

리뷰어는 아래 순서로 본다.

1. 동작이 맞는가
2. 회귀 가능성이 있는가
3. 구조가 현재 문서/정책과 어긋나는가
4. 테스트/문서가 충분한가
5. 그다음에야 표현/정리 수준을 본다

## 우선순위 기준

### P0

- 데이터 손실 가능성
- 인증/권한 문제
- 배포 장애
- 명백한 런타임 오류
- 잘못된 env 처리

### P1

- 사용자 흐름 회귀
- 잘못된 responsive 동작
- 접근성 저하
- 성능 저하 가능성
- 정보구조/라우팅 정책 위반

### P2

- 유지보수성 저하
- 문서 누락
- 네이밍/구조 일관성 문제
- 디자인 토큰/스타일 일관성 문제

## 리뷰 체크리스트

### 공통

- 변경 목적이 분명한가
- 파일 경계가 현재 레이어 규칙과 맞는가
- 불필요한 코드/문서/자산이 같이 남지 않았는가

### `apps/docs`

- `Feed / Web / Mobile / UI/UX / About / Docs` 정보구조와 맞는가
- shell breakpoint와 content breakpoint가 충돌하지 않는가
- `next-intl` 메시지 키 사용 규칙을 따르는가
- 로컬 MDX와 원격 HTML 경로 모두에서 회귀가 없는가
- `server-only` fetch와 클라이언트 렌더 경계가 맞는가

### 원격 콘텐츠 / 인증

- `BLOG_CONTENT_*`와 backend 계약이 맞는가
- `401/403` 처리 정책을 깨지 않았는가
- `slug / markdownPath / id` 규칙이 일관적인가
- 이미지/asset host 규칙이 현재 env 전략과 맞는가

### 디자인 / UI

- 디자인 토큰을 우선 사용했는가
- 라이트/다크 모드 모두에서 대비가 맞는가
- 모바일과 tablet 구간에서 의도치 않은 밀도 변화가 없는가

### 문서

- 관련 `worklog`가 추가/갱신됐는가
- 구조적 결정이면 `architecture` 문서가 필요한가
- 반복 운영 지식이면 `runbooks` 또는 `knowledge`로 승격할 가치가 있는가
- 이번 작업에서 새로 배운 점이 있다면, 해당 파트의 `docs/knowledge/*` 문서도 같이 갱신됐는가

## 리뷰 코멘트 작성 규칙

리뷰 코멘트는 아래 형식을 권장한다.

- `무엇이 문제인지`
- `왜 문제인지`
- `어느 조건에서 드러나는지`
- `어떤 방향으로 고치면 되는지`

좋은 예:

- `640px ~ 1023px 구간에서 카드 패딩이 sm에서 먼저 커져 shell 정책과 충돌합니다. responsive policy 기준상 내용 밀도 전환은 lg로 미루는 편이 맞습니다.`

## 머지 조건

아래 조건이 만족되면 머지를 고려한다.

- P0 미해결 없음
- 중요한 P1 이슈 해결 또는 합의된 후속 조치 기록
- 타입/테스트/최소 실행 확인 완료
- 관련 문서 업데이트 완료
- 반복될 운영 지식이 생긴 경우 `docs/knowledge/` 반영 완료

## 현재 `main` ruleset 의미

현재 저장소의 `main` 보호 규칙은 아래 의미로 해석한다.

### `Require a pull request before merging`

- `main`에 직접 push하지 않고, 모든 변경을 PR 단위로 남긴다.
- 혼자 개발이어도 변경 이유, diff, 문서 업데이트, AI 리뷰 기록을 PR에 묶어두는 것이 목적이다.

### `Required approvals: 0`

- 이 저장소는 현재 사람 승인 수를 merge 조건으로 강제하지 않는다.
- 혼자 개발 + AI 리뷰 흐름에서는 “사람 승인”보다 “PR 생성 + 셀프 리뷰 + CI 통과 + 문서 갱신”이 더 현실적인 품질 게이트다.

### `Require status checks`

- 현재는 실제로 생성되는 체크 이름만 required status check에 둔다.
- 의미는 “코멘트만 정리됐다고 merge하지 않고, 최소한 자동 검증이 통과한 변경만 main에 반영한다”는 것이다.
- 과거 체크 이름(`Build and Test`)을 ruleset에 남겨두면 PR이 `BLOCKED` 상태로 남을 수 있으므로, 현재 CI 이름과 항상 맞춘다.
- 현재 기준 권장 required status checks는 `Lint`, `Typecheck`, `Test`, `Analyze (actions)`, `Analyze (javascript-typescript)`이다.

### `Require conversation resolution`

- PR 대화나 리뷰 스레드가 열린 채로 merge하지 않는다.
- AI 리뷰 코멘트든 셀프 리뷰 메모든, 남겨진 쟁점은 해결하거나 명시적으로 정리한 뒤 merge하는 흐름을 의도한다.

### `Block force pushes`

- `main`의 이력을 덮어쓰지 않는다.
- 장기적으로 worklog, PR, commit 이력과 함께 문제를 추적할 수 있게 보존하는 목적이 있다.

## 이 저장소에서의 해석

즉 현재 ruleset 조합은 아래를 의미한다.

- 사람 승인 중심의 팀 리뷰 프로세스를 강제하는 저장소는 아니다.
- 대신 `PR 기반 변경 관리`, `자동 검증`, `대화 정리`, `이력 보존`을 main 보호 기준으로 삼는다.
- 품질 게이트는 앞으로도 “승인 수”보다 “명확한 기록과 재현 가능한 검증”을 우선한다.
